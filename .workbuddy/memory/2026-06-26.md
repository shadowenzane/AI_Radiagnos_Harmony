## v1.3.2 Bug 修复（2026-06-26 第三轮）

### Bug1：DeepSeek 测试连通性失败（连接不通）
**根因**：移动版 `LlmService.testConnectivity` 测试时发送 `'hi'` 并强制 `max_tokens=10`。DeepSeek 等模型在 `max_tokens` 过小时返回 400（insufficient_token），导致误判"连接不通"。实际诊断调用用 `max_tokens=4000` 能通，所以只有测试失败。
**对照桌面版**：`test_llm_connection` 用 `'请回复"连接成功"四个字'`，**不限制 max_tokens**，正常走 `_call_llm`。
**修复**（`llm_service.dart`）：测试消息改为桌面版文案；`max_tokens` 提到 100（gemini 的 maxOutputTokens 同样），不再压到 10。

### Bug2：火山方舟提示"API Key 为空"但配置页无此输入框
**根因**：上一轮把火山方舟简化为标准知识库（仅 AK/SK，无 api_key）后，`KnowledgeBaseService.query()`/`testConnectivity()` 的**最外层**仍统一检查 `credentials['api_key']` 为空即报错——火山方舟根本没有 api_key，必然触发。`diagnosis_service.dart` 第48行也用 `creds['api_key']` 过滤，会把火山方舟从诊断中排除。
**对照桌面版**：`test_kb_connection`（第254-262行）和 `_query_knowledge_base` 对 volcengine **特殊处理**：允许无 api_key，只要 `access_key+secret_key`。
**修复**：
- `knowledge_base_service.dart`：新增 `validateCredentials(type, creds)` 按类型校验（tencent 需 api_key+secret_key；volcengine 需 access_key+secret_key；notebooklm 需 api_key）。`query`/`testConnectivity` 外层用此方法替代统一 api_key 检查。
- `diagnosis_service.dart`：知识库凭证过滤改用 `KnowledgeBaseService.validateCredentials(kb.type, creds) == null`，火山方舟不再被误排除。

### 改进：知识库连通性测试误报"连接成功"
**问题**：`_testTencent`/`_testVolcengine` 用 `!warning.contains('网络')` 判断，配置不完整/签名失败时也误报成功。
**修复**：新增 `_evalQueryResultForTest` 共享方法，按 warning 文案精确分类：网络错误/配置不完整/签名失败/API错误 → 失败；"无匹配内容" → 成功。腾讯和火山方舟测试均复用。
